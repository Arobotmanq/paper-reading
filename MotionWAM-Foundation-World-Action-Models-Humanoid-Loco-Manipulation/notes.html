World Action Model WAM

World Action Model · 世界动作模型

📌 TL;DR

把视频生成器(世界模型)的动力学先验耦合进机器人策略：模型先"想象"未来会看到什么画面，再据此预测动作。比纯"图像-文本"的 VLA 多了一层对时间演化与接触物理的内在建模，行为更连贯、更物理可行。

1和 VLA 的区别

VLAWAM
底座静态图像-文本 VLM视频世界模型(视频生成器)
先验语义先验（认物体、懂指令）动力学先验（场景接下来怎么动）
动作怎么来直接回归 \(\pi(a|o,l)\)先 roll out 未来视觉状态，再反推动作
软肋缺时间/接触物理建模迭代去噪太慢，难实时

2在 MotionWAM 里

MotionWAM 是 WAM 的人形全身版：用 Video DiT 当世界模型、Motion DiT 当动作头。它解决了 WAM 长期的实时痛点——见 读中间去噪特征。综述 问题1 给出 WAM 慢的背景，结果6 证明它反超纯语义先验的 VLA。

🔗 关系

对比基线 Cosmos Policy 也是 WAM，但要迭代去噪完整未来视频（0.7Hz）；MotionWAM 只读中间特征（4.9Hz，快 7×）。WAM 范式全景见已读库 World Model 综述。

读中间去噪特征 本文核心

Intermediate Denoising Features · 单次前向读取生成过程的隐藏激活（DiT4DiT 思路）

📌 TL;DR

世界模型要把未来帧"画清楚"需多步迭代去噪（慢）。MotionWAM 不要画完的成品，只要它画到一半时某层的隐藏激活——把 Video DiT 固定在纯噪声端(τ≈1)跑"一次想象"，用 forward hook 截取单个 transformer block 的激活当条件。单次前向 = 实时。

1为什么这样能行

flow matching 生成是沿 \(\tau:1\to0\) 从噪声积分到清晰帧。关键观察：即使在 \(\tau\approx1\)（几乎全是噪声、还没开始去噪），网络为了预测速度场，其隐藏层已经编码了"场景接下来要往哪走"的语义+动力学信息。所以不必真的把帧去噪出来，读那一层激活就够给动作头当条件。

单次前向条件提取Eq.(2) 简化
z0_t        = VAE(当前第一视角帧)         # 干净条件
z_future    ~ N(0, I)                     # 未来帧 = 纯噪声 (τf ≈ 1)
h_t         = Hook[ v_video( z_future, τf | z0_t, l ) ]   # 单次前向, 读某 block 激活
# 从不迭代去噪 z_future —— h_t 直接喂给 Motion DiT 当条件

2效果

这单次前向是 MotionWAM 实时(4.9Hz)的命门：比同样基于世界模型、但要迭代去噪的 Cosmos Policy 快 7×（0.7Hz）。证据见综述 方法2 与 结果7。

🔗 关系

思路源自作者自家 DiT4DiT（联合建模视频动力学与动作）。它是 WAM 实时化的关键，依赖 DiT + flow matching 的生成机制。

DiT + Flow Matching 生成骨架

Diffusion Transformer + Flow Matching · 视频与动作的统一生成机制

📌 TL;DR

DiT = 用 Transformer 做骨干的扩散/流模型；flow matching = 训练它回归一个"噪声→数据"的连续速度场，推理时积分该场生成样本。MotionWAM 的 Video DiT 与 Motion DiT 都用这套，统一在一个 flow-matching 目标下训练。

1flow matching 一句话

给定干净样本 \(x_1\) 与噪声 \(\epsilon\)，沿直线 \(x_\tau=(1-\tau)x_1+\tau\epsilon\) 插值，网络回归速度场 \(v_\theta(x_\tau,\tau)\approx \epsilon-x_1\)。推理时从噪声解 ODE 积回 \(x_1\)。与 diffusion 同源，但目标更直接（回归速度而非逐步去噪分布）。

2MotionWAM 的两个 DiT

Video DiTMotion DiT
初始化Cosmos-Predict2.5-2B新挂，随机+每本体投影器
生成什么未来帧 latent 的速度场全身 motion latent 的速度场
损失\(L_\text{video}\)（式3）\(L_\text{motion}\)（式4，条件于 Video DiT 隐藏状态）
推理用法只跑一次前向取中间激活（不积分到底）正常积分出动作 token
🔗 关系

正因为生成是 flow-matching 的"沿 τ 积分"，才有 读中间特征的操作空间（在 τ≈1 截激活）。同款 flow-matching 动作头也用于 π0 系/OpenHLM——参见 OpenHLM 笔记的 flow matching 卡。

FSQ + SONIC 全身 token 动作接口

Finite Scalar Quantization + SONIC · 统一 whole-body motion latent

📌 TL;DR

SONIC 是个通用全身 motion-tracking 控制器，用单一共享 latent 融合各种运动目标。MotionWAM 用 FSQ 把这个共享 latent 压成 64 维离散 token（2 token × 32 级），概括移动/躯干/升降/脚部意图；再加手部连续通道。一个动作空间，腿和手平权。

1FSQ 是什么

Finite Scalar Quantization：VQ 的轻量替代——不学码本，而是把每个 latent 维直接四舍五入到固定的几个标量级别（如 32 级）。这里 2 个 token、每个 32 级，组合出 64 维离散向量 \(k_t\in\{-1,\dots,1\}^{64}\)。好处：无码本坍塌、实现简单、离散化稳定。

2为什么"解放了腿"

传统分层：上身收关节目标、下身只收 base 速度/高度/朝向 → 腿被锁成保平衡。MotionWAM 让全身共用同一套 motion token，腿因此拿到"任务性动作词表"，能踩踏板、踢球。推理时把连续标量 slot 最近邻取整成离散 index，交 SONIC 解码成关节命令 \(a_t\)。

动作解码链Eq.(6)
m_t = (m_cont, k̃)           # Motion DiT 输出: 连续末端通道 + 标量 token slot
   ──flow matching──►  m̂_t = (m̂_cont, k̂̃)
   ──k̂ = round(k̂̃)──►  (m̂_cont, k̂)
   ──SONIC 解码──►     a_t   # 全身关节命令
🔗 关系

SONIC 同样是 OpenHLM 用的低层控制器——两篇都把 SONIC 当全身执行底座，区别在高层（OpenHLM=VLA，MotionWAM=世界模型）。这套统一 token 由 Motion DiT 生成。
