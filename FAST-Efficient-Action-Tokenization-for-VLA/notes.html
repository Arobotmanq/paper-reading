Action Tokenization Core Problem

动作 Tokenization · 把连续机器人动作映射为离散 token 序列的过程

📌 TL;DR

自回归 Transformer 只能预测离散 token，但机器人动作是连续的关节角、末端速度等。Action tokenization 是这两者之间的桥梁——它的质量直接决定了 VLA 能不能学会高频灵巧任务。

1定义

给定一段 action chunk（H 步连续动作，每步 D 维），tokenization 的目标是定义一个映射：

\[ T_a: a_{1:H} \to [T_1, T_2, \ldots, T_n], \quad T_i \in \mathcal{V} \]

其中 \(|\mathcal{V}|\) 是词表大小（常用 256 或 1024），\(n\) 是 token 数量（可变长，类似文本 tokenization）。

2为什么 tokenization 是关键瓶颈

自回归 VLA 的训练目标是最大化 next-token prediction 的对数似然。学习信号正比于每个 token 相对于前序 token 的边际信息量。如果相邻 token 几乎一样（高频平滑动作），边际信息趋近于零 → 模型陷入"复制上一个 token"的局部最优。

3Naive Binning（先前主流做法）

对每个 action 维度独立分桶（通常 256 个 bin），逐时间步 tokenize：

\[ T_{a_{1:H}} = [T_{1,1}, \ldots, T_{1,D}, \ldots, T_{H,1}, \ldots, T_{H,D}] \]
  • 50Hz、14 维动作 → 700 tokens/秒，且相邻 token 几乎完全相同
  • RT-2、OpenVLA 均用此方案，在低频任务（BridgeV2 5Hz）还能用，高频任务（DROID 15Hz、T-shirt 50Hz）完全失效

4FAST 的改进

先用 DCT 把时间相关性转化为频域稀疏性，再用 BPE 去除冗余，使每个 token 都携带真正新颖的信息。见 FAST Tokenizer 词条。

📜 出处

Naive binning 最早用于 RT-1（2022）、RT-2（2023），被 OpenVLA（2024）继承。FAST 是 2025 年 1 月提出的改进方案，已成为 π0.5/openpi 生态的默认选择。

DCT Discrete Cosine Transform

离散余弦变换 · FAST 的核心压缩算法

📌 TL;DR

DCT 是 JPEG 图像压缩的底层算法——把空间像素转成频率系数，平滑区域只需几个低频系数就能表示。FAST 把相同思路用于时间序列：把高度相关的连续动作时序转换为稀疏的频率系数，让多余的高频零系数可以被直接丢掉。

1数学定义

对长度 H 的一维信号 \(x = [x_0, x_1, \ldots, x_{H-1}]\)，Type-II DCT 定义为：

\[ C_k = \sum_{n=0}^{H-1} x_n \cos\!\left[\frac{\pi}{H}\left(n + \frac{1}{2}\right)k\right], \quad k = 0, 1, \ldots, H-1 \]
  • \(C_0\)：直流分量（均值）
  • \(C_1, C_2, \ldots\)：越来越高频的振荡分量
  • 平滑信号（如机器人关节角轨迹）：能量集中在低频，\(C_k \approx 0\) for large \(k\)
🎵 类比：音乐的频谱

就像音乐可以用 wav（时域）或 mp3（频域）存储，平滑的钢琴曲 mp3 极小但 wav 极大。机器人抓取动作就是"钢琴曲"——用频域表示，几个系数就够了。

2在 FAST 中的用法

对 D 维 action chunk，每个维度 \(i\) 独立做 DCT：

FAST DCT Step伪代码
for i in range(D):          # 每个 action 维度独立处理
    C[i] = DCT(a[i, 1:H])  # 得到 H 个频率系数
    C_q[i] = round(γ · C[i])  # 量化，γ=10；大部分高频为 0
  • 结果：D×H 稀疏矩阵，多数元素为 0，少数低频系数非零
  • 量化尺度 γ=10 是全局超参，不需要数据集特定调优
  • 逆变换 IDCT 用于解码 → 重建连续动作

3为什么选 DCT 而不是 FFT 或学习压缩

方法优点缺点
DCT（FAST 选择）纯分析方法，无需训练；超参少且不敏感；可逆；实现极简非自适应（但实验显示够用）
FFT（复数变换）标准信号处理输出复数，tokenization 不方便
VQ-VAE / FSQ（学习压缩）自适应，可能更优需要单独训练；超参敏感；实验显示 FAST 性能更好

4出处

📜 起源

DCT 由 Ahmed, Natarajan, Rao 1974 年提出，后被 JPEG（1992）采用为图像压缩标准。FAST 是第一个将 DCT 用于机器人 action tokenization 的工作。

BPE Byte Pair Encoding

字节对编码 · FAST 的无损压缩组件

📌 TL;DR

BPE 是 GPT、LLaMA 等 LLM 的文本 tokenizer 核心算法：把频繁出现的符号对合并成一个新符号，反复迭代直到达到目标词表大小。FAST 用 BPE 来"消化"DCT 后的稀疏整数序列，把大量 0 和常见系数组合合并，实现无损压缩。

1BPE 训练过程

BPE 训练算法伪代码
初始词表 = {所有出现的整数值}
训练数据 = DCT 量化后的所有序列（展平为 1D 整数序列）

while len(词表) < 目标大小:
    找到相邻符号对 (a, b) 出现频率最高者
    将 (a, b) 合并为新符号 c，更新词表和训练数据

输出：BPE 词表 Φ（合并规则字典）

2在 FAST 中的具体作用

DCT 量化后的序列大量包含：

  • 大量 0：高频系数量化后归零 → BPE 把连续的 "0, 0, 0, ..." 合并成一个 token
  • 常见低频系数组合：不同维度的低频系数经常以相似的模式出现 → BPE 把高频组合直接映射为单个 token

最终效果：50Hz 双臂 T-shirt folding 从 700 个 naive tokens → 53 个 FAST tokens（13.2× 压缩）。

3与语言模型 BPE 的异同

语言 BPE（GPT 等）FAST 中的 BPE
输入序列字符/字节序列DCT 量化后的整数序列
训练数据大规模文本语料机器人 action 轨迹（FAST+：1M 轨迹）
词表大小~50k（GPT-2）~128k（GPT-4）1024（可接入 VLM 原有词表）
目的减少序列长度、表示罕见词消除零值、合并常见系数组合

4为什么 BPE 不可缺

FAST 消融实验（§VI-D）显示：去掉 BPE、只保留 DCT，策略性能明显下降——不仅因为大量 0-token 稀释学习信号，还因为序列变长（数百个 token）导致自回归推理极慢。BPE 是 FAST 的效率保障。

📜 出处

BPE 最初是数据压缩算法（Philip Gage, 1994）。Sennrich et al. 2016 将其引入 NLP 文本 tokenization，后被 OpenAI GPT 系列广泛采用（tiktoken）。FAST 是将 BPE 用于机器人 action 序列压缩的首次尝试。

FAST / FAST+ The Tokenizer

Frequency-space Action Sequence Tokenization · 频域动作序列 Tokenization

📌 TL;DR

FAST = DCT + BPE 的完整 action tokenization 算法（针对单个数据集训练 BPE 词表）。FAST+ = 在 100 万条真实机器人轨迹上预训练的通用版，可黑盒插用于任意新 robot/任务，三行 HuggingFace 代码调用。

1完整算法流程

FAST Tokenizer（Algorithm 1）伪代码
输入: action chunk a[1:H]（H步，D维），超参 γ=10

1. 归一化: 将每个维度映射到 [-1, 1]（用 1st/99th 分位数）
2. DCT: 对每个维度 i 独立计算 DCT 系数 C[i] = DCT(a[i, 1:H])
3. 量化: C_q[i][j] = round(γ · C[i][j])   # 大部分高频系数变 0
4. 列优先展平: [C_q[1][1], C_q[2][1], ..., C_q[D][1],   # 所有维度最低频
                C_q[1][2], C_q[2][2], ..., C_q[D][2],   # 所有维度次低频
                ...]
5. BPE 压缩: tokens = BPE(展平序列, 词表 Φ)
返回: token 序列 tokens（约 30 个/臂，与频率无关）

解码(反向): tokens → BPE解码 → 整形为矩阵 → IDCT → 连续动作

2关键设计决策：列优先展平

展平方式有两种选择：

  • 列优先（FAST 选择）：低频分量优先——先预测所有维度的低频（整体运动方向），再预测高频（细节调整）
  • 行优先：完整预测第一个维度的所有频率，再预测第二个维度

实验表明列优先更稳定——类比语言模型先确定句子主题再填细节的认知顺序。

3FAST vs FAST+

FASTFAST+
BPE 词表训练数据目标数据集（需要几分钟重新训练）1M 跨 embodiment 轨迹（预训练好）
使用门槛需要访问目标数据集三行代码，开箱即用
性能数据集专属最优与 FAST 持平（实验验证）

4与 VLM 词表的集成

FAST token 被直接插入 VLM 原有词表——用最少使用的 token slot 替换为 action token（与 OpenVLA 相同策略）。这意味着不需要修改 Transformer 架构，任何自回归 VLM 都可以直接使用 FAST。

Action Chunking Temporal Abstraction

动作块预测 · 一次预测多步连续动作而非单步

📌 TL;DR

策略不预测单个时间步的动作，而是一次预测未来 H 步（1 秒）的完整动作序列。这减少了复合误差、提高了时间一致性，但也让 tokenization 更具挑战性——一个 chunk 里相邻动作高度相关。

1为什么用 Action Chunking

  • 减少复合误差：单步预测的误差会在执行时累积；chunk 预测提前规划整段动作，误差不传播。
  • 时间一致性：一次输出连贯的动作序列，避免帧间抖动。
  • 支持高频控制：模型以较低频率（如 1Hz）预测 chunk，实际以高频率（50Hz）执行，解耦推理频率与控制频率。

2在 FAST 中的关系

FAST 的输入就是 1 秒的 action chunk。Action chunking 带来的时间相关性（相邻步动作几乎相同）正是 naive tokenization 失败的根本原因，也是 DCT 压缩能大幅减少 token 数的根本原因——平滑的 chunk 在频域只需少数低频系数。

📜 出处

ACT（Chi et al. 2023, Diffusion Policy）将 action chunking 推广到机器人学习。FAST 论文中 H（chunk 长度）取 1 秒，即在各数据集控制频率下的对应步数（15-50 步）。

自回归 vs Flow Matching VLA Decoding

两大 VLA action 解码范式的权衡对比

📌 TL;DR

自回归（AR）VLA 把 action 当作离散 token 顺序生成，像写文字一样；Flow matching（扩散）VLA 把 action 当作从噪声去噪的连续向量。两者性能相近，AR 训练快 5× 但推理慢 7.5×，语言跟随更好；扩散推理快但训练慢、语言跟随略差。

1详细对比

维度自回归（AR）+ FASTFlow Matching（扩散）
训练范式Next-token prediction（离散 CE loss）Flow matching / diffusion（MSE loss on noise）
Action 表示离散 token 序列（~30-60 个）连续向量（经 300M action expert 预测）
推理时间~750ms/chunk（4090 GPU）~100ms/chunk（10 步去噪）
训练速度5× 更快基准（需要更多 GPU hours）
语言跟随更好（DROID 实验证明）略差（扩散模型有时忽略语言指令）
大数据收敛更快（Table Bussing 3× fewer steps）需要更多步骤
适合场景训练成本敏感 / 语言跟随要求高 / 静态操作推理实时性要求高 / 动态任务

2π0 的混合策略（π0.5 预训练）

π0.5 预训练阶段用 FAST 离散 token（充分利用 VLM 预训练语言能力），后训练阶段加入 flow matching action expert（提高推理速度和连续精度）——兼取两者之长。这也解释了为什么 FAST 是 π0.5 的重要基础。

💡 对你项目的启示

如果训练 VLA 的 GPU 时间是主要约束 → 选 AR + FAST。如果推理实时性（50Hz 闭环）是硬约束 → 选 flow matching，或等 FAST 的推理加速工作。π0.5 的混合方案可能是未来的最佳实践。

VLA Vision-Language-Action

视觉-语言-动作模型 · FAST 的应用目标

📌 TL;DR

VLA = 在 VLM（视觉-语言预训练模型）上追加机器人动作预测能力的端到端策略模型。给一张图片 + 一条语言指令 → 直接输出机器人要执行的动作序列。FAST 解决的是"动作如何表示为 VLM 能处理的 token"这个核心问题。

1架构结构

[图像帧] → 视觉编码器（如 SigLIP）→ 视觉 token
[语言指令] → 语言 token
[历史动作/状态] → 动作/状态 token

全部拼接 → Transformer（如 PaliGemma、Gemma）→ 预测下一个 token

如果是 AR VLA：输出 action token 序列 → FAST 解码 → 连续动作
如果是 diffusion VLA：输出触发 action expert → 扩散去噪 → 连续动作

2FAST 在 VLA 中的位置

FAST 处于 VLA 的"输入/输出接口"位置：训练时把连续 action chunk 编码为 token 加入训练序列；推理时把模型输出的 token 解码回连续动作。它对 Transformer 架构本身无任何修改要求，可即插即用于任何自回归 VLM。

Universal Tokenizer Cross-Embodiment

通用 Tokenizer · 跨 embodiment / 数据集的预训练 tokenizer

📌 TL;DR

FAST 的 BPE 词表理论上需要对每个新数据集重新训练（几分钟，但有额外摩擦）。FAST+ 通过在大规模跨 embodiment 数据集上预训练一次 BPE 词表，使其可以直接用于任意新机器人/任务——就像 LLM 的 tokenizer 无需对新文档重新训练一样。

1FAST+ 训练数据

约 100 万条 1 秒 action chunks，覆盖：

  • 单臂（UR5 等）、双臂（ARX、Trossen）、移动操作机器人
  • 关节控制（joint）和末端执行器控制（end-effector）两种 action space
  • 5Hz（BridgeV2）到 50Hz（T-shirt folding）多种控制频率
  • 包含 BRIDGE v2、DROID、OXE 等公开数据集 + Physical Intelligence 自采数据

2泛化性能

在 完全未出现在 tokenizer 训练集的新数据集上（包括 NYU 灵巧手、UMI、人形机器人 HumanoidPlus、甚至 Waymo 导航数据），FAST+ 相比 naive tokenization 仍有 2× 以上压缩率，且策略训练性能与数据集专属 FAST 持平（Figure 6, 8）。

3使用方法（三行代码）

HuggingFace AutoProcessorPython
from transformers import AutoProcessor

tokenizer = AutoProcessor.from_pretrained(
    "physical-intelligence/fast",
    trust_remote_code=True
)

# 编码（推荐先归一化到 [-1, 1]）
tokens = tokenizer(action_chunk)  # action_chunk: np.ndarray [H, D]

# 也可在自己的数据集上快速训练新词表：
new_tokenizer = tokenizer.fit(action_dataset)
💡 实践建议

直接使用 physical-intelligence/fast 作为默认 tokenizer；只有当你的 action space 极其特殊（如高频 >50Hz 全身人形控制）且有大量数据时，才值得 tokenizer.fit() 训练数据集专属版本。
