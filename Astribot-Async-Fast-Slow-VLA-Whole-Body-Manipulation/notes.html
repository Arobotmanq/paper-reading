异步快慢 VLA Dual System

Asynchronous Fast-Slow VLA · 真并行快慢双系统 VLA

📌 TL;DR

慢路（大 VLM，思考）和快路（轻量 action head，行动）完全在不同线程/频率下并行运行，靠一个共享 buffer 交换信息——快路不等慢路，控制频率只受快路自身速度限制。

1定义与背景

灵感来自 Kahneman 的"系统1（快速直觉）/ 系统2（慢速推理）"认知理论。在 VLA 里：

路径模型频率负责什么
慢路（System 2）大 VLM（3–7B）1–3 Hz语义理解、推理、子任务规划、生成粗粒度 action token
快路（System 1）轻量 diffusion policy25–30 Hz实时连续动作生成，响应当前传感器输入

同步版（多数现有方法）：快路每步都等慢路完成 → 整体频率 = 慢路频率（~3–12 Hz）。
异步版（DuoCore-FS）：两路完全并行，快路从 bridge buffer 读最新语义，不等慢路 → 整体频率 = 快路频率（~30 Hz）。

2在 DuoCore-FS 里的具体实现

慢路（PaliGemma-3B）：以 1–3 Hz 解析多模态输入，生成 text embedding + fusion query embedding + reasoning token，写入 bridge buffer。快路（Pi0-small 风格 diffusion policy decoder）：以 25–30 Hz 从 buffer 拉取最新语义，与当前视觉/本体感知融合，经 Transformer encoder → diffusion 去噪生成 action chunk（T=32 步）。两路端到端联合训练，慢路 loss 权重 λ=1，快路 loss 权重 λ=10。

3与其他异步实现的对比

系统异步方式慢路输出端到端训练开源
FiS-VLA / OpenHelix固定调度比（非真并行）latent是是
Helix (Figure AI)真并行仅 latent（无 action token）未知否
Hume真并行（级联架构）latent + action token否是
DuoCore-FS真并行latent + action token + reasoning是商业闭源

4类比

🎯 日常类比

像驾驶时的"导航系统"（慢，每隔几秒重算路线）和"驾驶反射"（快，随时踩刹车/转方向盘）：导航不需要每帧更新，驾驶动作必须实时响应。两者靠"当前导航指令"这个共享状态沟通，互不阻塞。

5出处

理论根源：Kahneman (2011) "Thinking, Fast and Slow"；机器人应用由 Bonnefon & Rahwan (2020) "Machine thinking, fast and slow" 引入讨论；VLA 实践由 π0（Physical Intelligence, 2024）率先实现双系统，DuoCore-FS 将其推进到真异步。

Bridge Buffer Async Interface

异步语义缓冲区 · 快慢路解耦接口

📌 TL;DR

一个内存缓冲区，慢路定期往里写"语义快照"（text embedding + fusion query + reasoning token），快路随时读最新快照来条件化动作生成——这是两路运行在不同频率却能协作的关键机制。

1存储内容

Bridge buffer \(B_t\) 在每次慢路更新后存储三类表示：

  • instruction embeddings：原始任务指令的 text token embedding，固定不变，防止 fusion query 过度专注于动作模式导致语言跟随能力退化。
  • fusion query embeddings：由可学习 query \(q_\psi\) 跨视觉/语言/本体感知/VLM 内部语义空间聚合而成，参数 ψ 由快路 action loss 驱动训练。
  • reasoning token：VLM 生成的 chain-of-thought 或子任务指示符等高层推理输出。

2为什么要同时保留 instruction embedding

论文的关键工程洞察：fusion query 的优化目标是动作质量，训练时可能"遗忘"与当前动作无直接关联的任务细节（比如"关门"vs"拿杯子"的区别）。保留原始 instruction embedding 作为稳定的语言约束，防止这种退化——这是 DuoCore-FS 相比单纯 latent buffer 设计的重要差异。

3可微性

Buffer 作为可微接口实现端到端训练：快路的 action loss 梯度通过 buffer 流回慢路的 fusion query 参数 ψ。这使 fusion query 的内容直接为控制目标服务，而不只是语义重建。

4延迟容忍

工程细节

快路不需要等慢路更新——若慢路还未产生新输出，快路使用上一次写入的 buffer 内容继续出动作。这使得即便慢路临时卡顿（例如推理耗时略超预期），快路也能平滑地持续运行，不出现"等待卡顿"。论文实验中 ∆t 的时间延迟在训练时随机采样 U[0, 25] 帧，覆盖部署时的真实延迟范围。

5类比

类似多线程编程中的 shared memory + producer-consumer 模式：慢路是 producer（低频写），快路是 consumer（高频读）。Bridge buffer 相当于一个有锁（或无锁读写安全）的内存区域，consumer 总能读到最新的 producer 输出，不会阻塞。

VLA Policy

Vision-Language-Action Model · 视觉-语言-动作模型

📌 TL;DR

VLA 是把"看图"、"理解指令"、"输出动作"三件事统一进一个端到端模型的机器人策略——核心思路是把 LLM/VLM 的预训练跨模态知识迁移到机器人控制上。

1定义

VLA 以视觉观测 \(\{I_t^1, ..., I_t^n\}\)、本体感知 \(q_t\)（关节角/速度）和自然语言指令 \(l\) 为输入，直接输出连续动作 \(a_t\)（关节角度/末端位姿/速度命令等），中间不需要手工设计的感知模块或规划器。

2两类主流架构

类型代表动作生成方式优点缺点
自回归 token 预测RT-2, OpenVLA, π0-FAST离散 action token 逐步预测天然接 VLM 接口；支持 chain-of-thought频率低（~1–3 Hz）；精度受限于离散化
扩散/流匹配π0, GR00T N1, DuoCore-FS 快路连续 action chunk 并行去噪高频（~25–30 Hz）；连续高精度难与 VLM 自回归接口直接融合

DuoCore-FS 的设计选择：慢路用自回归（VLM 出 action token/reasoning），快路用扩散策略（连续高频出 action chunk），两者异步协作，兼取双方优点。

3在 DuoCore-FS 里

慢路 VLM（PaliGemma-3B）对应自回归 VLA，优化目标为 \(\mathcal{L}_{slow}(\phi) = \mathbb{E}_{(o_t, l, r_t) \sim D} [-\log p(r_t | o_t, l)]\)（NLL 损失）。快路 diffusion policy 对应流匹配 VLA，优化目标为对去噪向量场的 MSE。

4出处

RT-1（Google Robotics, 2023）首次大规模 VLA；RT-2（Google, 2023）用 VLM 预训练权重直接迁移；π0（Physical Intelligence, 2024）引入 flow matching action expert，DuoCore-FS 的快路即以其为基础（Pi0-small 风格）。

全身动作 Tokenizer RVQ-VAE

Whole-Body Action Tokenizer · 几何感知残差向量量化自编码器

📌 TL;DR

把全身 25 DoF 的连续动作 chunk 压缩成固定长度离散 token 序列，让慢路的自回归 VLM 能高效预测全身动作——关键设计：按 position/rotation/gripper 三路分开量化，rotation 用 SO(3) 测地距离，避免 FAST 的组合爆炸。

1为什么全身操作需要专门的 tokenizer

固定臂 VLA（如 π0-FAST）的 tokenizer 是为低维（~7 DoF）动作设计的。全身 25 DoF 动作在时间维度展开后，FAST tokenizer 平均产生 81.12 token（最长 205 token），速度仅 0.95 Hz，且因顺序预测左手/右手/躯干产生误差累积，成功率降为 0%。DuoCore-FS 的 RVQ-VAE tokenizer 固定输出 36 token，速度 3.27 Hz，慢路成功率 83.3%。

2三路分解结构

动作 \(a_t \in \mathbb{R}^{29}\) 分解为：

  • position \(P \in \mathbb{R}^{T \times 9}\)：躯干和两个末端执行器的 Cartesian 位置增量（各 3D），用 \(\ell_2\) 重建 loss。
  • rotation \(R \in \mathbb{R}^{T \times 18}\)：躯干和两个末端的 SO(3) 姿态增量（6D 连续表示），用 SO(3) 测地距离 loss \(\ell_{geo}\)——比欧氏距离更几何正确。
  • gripper \(G \in \mathbb{R}^{T \times 2}\)：左右夹爪绝对开合宽度（非增量），用 \(\ell_2\) 重建 loss。

每路各有独立 1D-conv encoder-decoder + 残差 VQ（RVQ）栈，codebook 大小 1024。

3RVQ（残差向量量化）

普通 VQ 只用一本 codebook，RVQ 叠加多级：第一级量化残差后，第二级再对剩余残差量化……逐级压缩，在相同 codebook 大小下表达能力远强于单级 VQ。在 DuoCore-FS 里每个 RVQ 栈的 codebook 大小为 1024，训练时包含 commitment loss + codebook loss（标准 VQ 目标）。

4为什么 rotation 要用测地距离

SO(3) 空间不是欧氏空间——两个旋转矩阵的欧氏距离不能正确反映旋转差异（例如两个几乎相反的旋转欧氏距离可能很小，但测地距离很大）。使用 \(\ell_{geo}\)（即旋转轴角的角度）作为 loss 能让模型学到几何正确的旋转插值和预测。

与 FAST tokenizer 的对比（Table 5 数据）

FAST：平均 81.12 token / max 205 token / 成功率 0% / 频率 0.95 Hz
RVQ-VAE：固定 36 token / 成功率 83.3% / 频率 3.27 Hz
结论：FAST 直接用于全身操作不可行；RVQ-VAE 是全身 VLA 的必要设计而非优化项。

Action Chunking Control

动作分块 · 短时程动作序列预测

📌 TL;DR

不预测单步动作，而是一次预测一段（chunk）连续动作序列——避免模型在高频控制中反复"想"，减小延迟，也让模型能学到更平滑的动作规律。

1定义

设 action chunk 长度为 T，则模型在时刻 t 一次性预测 \(A_{t:t+T-1} = [a_t, a_{t+1}, ..., a_{t+T-1}]\)，而非只预测 \(a_t\)。执行时逐步发送，下一个 chunk 在当前 chunk 执行完前就已预测好（可流水线）。

2在 DuoCore-FS 里

快路扩散 policy 的训练目标为整个 chunk \(A_{t:t+T-1}\)，chunk 长度 T=32。在流匹配框架下，对 chunk 采样高斯噪声 \(\varepsilon\)，构造 \(X_\tau = \tau \varepsilon + (1-\tau) A_{t:t+T-1}\)，训练去噪向量场 \(v_\theta\)。全身动作 tokenizer 也以 chunk 为单位运作。

3优势

  • 减少延迟感知：每 T 步只需调用一次模型，控制频率等效提升。
  • 学习时间相关结构：模型可以学到连续动作的平滑性、速度曲线等时序特性。
  • 适合扩散/流匹配：并行去噪天然输出序列，比逐步自回归更快。

4出处

Action chunking 由 ACT（Chi et al., 2023）引入机器人模仿学习；Diffusion Policy（Chi et al., 2023）将其与扩散模型结合；π0 进一步用 flow matching + chunk 实现高频实时控制。

VLM Backbone Architecture

Vision-Language Model Backbone · 视觉语言模型骨干网络

📌 TL;DR

VLM backbone 是 VLA 慢路的"大脑"——一个在海量图文数据上预训练好的多模态基础模型，提供强大的视觉语义理解和语言推理能力，VLA 在其上做 fine-tuning 以获得操作能力。

1定义

VLM 以图像（经 vision encoder，如 SigLIP）和文本 token 为输入，经大型语言模型 backbone（如 Gemma、Qwen2.5、PaliGemma）生成文本或 token 输出。它的预训练数据来自网络图文对，具备 zero-shot 视觉问答、目标定位、推理等能力。

2在 DuoCore-FS 里

论文用 PaliGemma-3B（SigLIP vision encoder + Gemma-3B LLM）作为慢路 VLM，并基于 π0-FAST 的预训练权重初始化（π0-FAST 是在 PaliGemma-3B 上 fine-tune 的 VLA）。论文也提到可替换为 Qwen2.5-VL-3B 或 Qwen2.5-VL-7B，框架灵活。

3规模与频率的权衡

VLM 规模推理延迟语义质量适用场景
~3B（PaliGemma-3B）~300–1000ms好实时操作（1–3 Hz 慢路）
~7B（Qwen2.5-VL-7B）~1–3s更好低频更新场景，或更强 GPU
~70B+（GPT-4V 等）~5–30s最好离线规划，无法实时部署

DuoCore-FS 的异步设计正是让 3B VLM 以 1–3 Hz 运行成为可接受的——因为快路不等它，只需要它定期更新 buffer 即可。

4加速手段：Jacobi 并行解码

论文采用 Jacobi-style parallel decoding（启发自 CLLMs，Consistency Large Language Models）：在自回归生成时，允许多个 token 位置并行迭代精化，而非严格顺序解码。这进一步降低了慢路每次推理的实际延迟，提升慢路更新频率。

5出处

PaliGemma 由 Google DeepMind 2024 发布（SigLIP + Gemma）；Qwen2.5-VL 由阿里巴巴发布；Jacobi 解码策略来自 CLLMs（Kou et al., 2024）。
