核心概念详解

Flow Matching Action Head

Flow Matching · 流匹配动作生成（π0 系动作头）

📌 TL;DR

把"从噪声生成一段动作"建模成学一个连续向量场，推理时从随机噪声出发、沿该向量场多步积分（典型 10 步）把噪声推成动作 chunk。π0 / π0.5 / OpenHLM 的动作头都是它。

1概念

和 diffusion 同属"生成式动作头"思路，但训练目标更直接：给定噪声 \(x_0\) 与目标动作 \(x_1\)，沿直线插值 \(x_t=(1-t)x_0+t\,x_1\)，网络回归该路径上的速度场 \(v_\theta(x_t,t)\approx x_1-x_0\)。推理时解 ODE \(\dot x = v_\theta\) 从 \(t{=}0\) 积到 \(t{=}1\)。

"多步"指 ODE 积分用多少个离散步——步数越多轨迹越平滑、延迟越高；步数越少越快、但动作可能更抖。

2在 OpenHLM 里干什么

π0.5 自带 flow-matching 头，OpenHLM 原样保留：10 步去噪，单次推理 ~90ms（5080 GPU）。作者试过把它换成单步以提速到 ~60ms，结论是不划算：

动作生成验证 action MSE真机 task progress推理延迟
10 步 flow matching（保留）≈0.009基线~90ms
1 步 flow matching≈0.007（更低）低约 20 点~60ms
drifting model（单步）≈0.007（更低）低约 20 点~60ms

这正是综述 洞察5 的第二个证据：MSE 更低 ≠ 真机更好。作者猜单步产出的动作 ℓ2 上接近、但时间上更抖、不够平滑。

🔗 关系

与 遥操接口无关、与 低层控制器互补：flow matching 在高层产参考动作，motion tracker 在低层把它跟踪到关节。对比路线见综述里 FAST（离散动作 token）。

Motion-Tracking 控制器 Low-Level Control

通用全身 motion tracker（SONIC）+ 重定向（GMR）+ 预览延迟

📌 TL;DR

两级层级控制的"下层"：吃高层 10Hz 的全身参考命令，用一个 RL 训练的通用 motion tracker 在 50Hz 输出目标关节位置，再由 PD 跟踪。OpenHLM 用 SONIC 当 tracker、GMR 当重定向器。

1三个零件

  • GMR（General Motion Retargeting）：把人体/SMPL 动作实时重定向到 Unitree G1 的每个关节，解决"人和机器人骨架不同"。
  • SONIC：一个被"超大规模"训练的通用 motion-tracking 控制器，能跟踪几乎任意全身关节轨迹，保持平衡。OpenHLM 直接复用它当低层。
  • PD 控制器：最底层，把目标关节位置变成力矩。

2preview latency Δt（本文重点调参）

motion-tracking 控制器可调一个预览延迟 Δt：它提前看到多远的未来参考动作再跟踪。看得越远→动作越平滑，但操作员指令到执行之间的延迟越大。

Δt现象Cola 放置 task progress
0 s最跟手，但行走顿挫/跺脚60%
0.2 s（采用）响应与平滑兼顾，演示时长几乎不变67%
0.4 s开始累积延迟60%

🔗 关系

是 遥操接口的执行底座：joint-based 遥操采到的关节轨迹，正是交给这个 tracker 跟踪的。对 Track B（足式控制）最相关——属于 locomotion / 全身控制范畴。

全身遥操三接口 Data Collection

joint-based / SMPL-based / VR 3-point（+ decoupled）

📌 TL;DR

遥操接口决定能采到什么演示，进而决定策略能学到什么。关键差异是暴露多少自由度：只暴露部分自由度的接口，有些任务"结构上做不到"。OpenHLM 选 joint-based 全身遥操。

1四种接口对比

接口动作维度下身怎么来能踩踏板/下蹲？
Decoupled 解耦21RL 控制器跟 base 速度+根高否（下身只是底盘）
VR 3-point24学出的运动规划器生成下身否
Joint-based 全身（采用）32动捕→GMR 重定向到每个关节是
SMPL-based81SONIC 原生吃 SMPL，省在线重定向是，但维度冗余更难学

2两个结论

  • joint-based 唯一全通：三任务（含取杯下蹲、丢瓶脚踩）全完成；解耦/VR 有任务结构上做不到（综述 洞察2 Table 1）。
  • 在线 joint-space retarget 击败原生 SMPL：88% vs 75%——SMPL 的 81 维里很多是给定运动链后的冗余维，VLA 仍得全学，反成更难的学习问题。
🔗 关系

采到的轨迹交给 motion-tracking 控制器执行；与 HuMI 互补——HuMI 是"无机器人"的廉价补充，遥操是"有机器人"的高质量来源。

HuMI Robot-Free Data

Humanoid Manipulation Interface · 人形版 UMI（机器人无关采集）

📌 TL;DR

UMI（手持夹爪在野外采操作数据，无需机器人）的人形版：两个手持 UMI 夹爪 + 三个体态 tracker（骨盆+双脚），采到任务空间 SE(3) 轨迹，再用 IK 抬成全身关节位置。采集快、便宜，但当前只能"教语义、教不了运动"。

1为什么便宜

机器人不在回路里：无需 reset 机器人、不会过热停机。每个 held-out 任务 40 条演示——HuMI 7min / 同体静止遥操 13min / 全身遥操 21min。且 HuMI 动作维度与语义直接对齐 joint-based 全身遥操，co-training 只需混流、无需改架构。

2为什么只传语义不传运动

HuMI 与真机遥操有两轴域差：

  • 视觉：相机不同（RealSense vs 校正后的 GoPro 鱼眼）、夹爪不同（机器人自适应平行夹 vs HuMI 刚性手持夹）。
  • 动作：HuMI 本质是人体动作，即便 IK 重定向后，轨迹仍不同于直接遥操机器人产生的轨迹（没有机器人在回路）。

当前数据量下，策略主要把 HuMI 当语义监督（认新物体、接地新指令），学不到全新运动（如 Pouring 倒水的倾杯）。作者推测放大 HuMI 数据量可能解锁运动迁移。

🔗 关系

是综述 方法6 的核心：与同体静止遥操对照——静止遥操供"运动+语义"，HuMI 只供"语义"但成本减半。HuMI 本身是作者自家工作（arXiv:2602.06643）。同方向参见 HoMMI（从人类演示学全身移动操作）。
