OVMM Task Definition

Open-Vocabulary Mobile Manipulation · 开放词汇移动操作

📌 TL;DR

OVMM 是"把任意对象从任意起始位置移到任意指定位置"的任务形式，核心难点是对象和场景都是未见过的（open-vocabulary），需要同时整合感知、语言理解、导航和操作四个子系统。

1概念定义

OVMM 任务的指令格式统一为：

"Move (object) from the (start_receptacle) to the (goal_receptacle)."
  • object：小型可操作家用对象（杯子、毛绒玩具、盒子等），测试时全为未见实例（unseen instances）。
  • start_receptacle / goal_receptacle：大型家具（桌子、椅子、沙发等），类别全部已见，但位置未见。
  • open-vocabulary：测试时对象来自训练时未见过的类别（unseen categories），机器人不能死记对象外观。

2四阶段评分协议

成功被分解为四个阶段，每个阶段都有部分成功分：

阶段目标仿真判定条件
FindObj找到起始接收器上的目标对象距离 ≤0.1m 且对象占视野 ≥0.1%
Pick抓取目标对象FindObj 成功 + arm 到达对象 0.8m 内（仿真 magic snap）
FindRec找到目标接收器Pick 成功 + 距目标接收器 ≤0.1m
Place将对象放置到目标接收器上FindRec 成功 + 对象稳定停留在接收器上 50 步

整体成功 = 所有四个阶段均成功（仿真 1250 步内，真实 300 步内）。部分成功指标 = 四个阶段成功率的平均值，作为并列排名时的 tiebreaker。

3为什么"开放词汇"是核心难点

传统操作基准（如 YCB 对象集）使用固定对象集，机器人可以用模板匹配识别。OVMM 要求在测试时处理训练时从未见过的对象类别（如 Rubik's Cube、Toy Drill、Lemon），这要求感知系统能够基于语言描述定位任意对象——这正是 CLIP、DETIC 等视觉-语言模型的用武之地。

5类比

🎯 直觉类比

想象你第一次进到一个朋友家，被要求"把厨房台面上那个绿色东西放到餐桌上"。你从未见过这个特定的绿色物体，也不知道这家的厨房布局——但你能用语言理解 + 视觉搜索 + 导航 + 操作完成任务。这就是 OVMM 要求机器人做的事。

Habitat Simulator

AI Habitat · 具身 AI 仿真平台（Facebook/Meta AI Research）

📌 TL;DR

Habitat 是 Meta AI 开发的高效 3D 具身 AI 仿真平台，支持在大量逼真室内场景中训练导航、操作策略。HomeRobot 的仿真组件就运行在 Habitat 之上，使用 HSSD 场景数据集。

1平台背景

Habitat 由 Savva et al. 在 ICCV 2019 首次发布，此后经历 Habitat 2.0（NeurIPS 2021，加入操作支持）等版本迭代。核心优势是渲染速度极快（可以在单 GPU 上并行运行数百个仿真环境），非常适合 RL 训练。

2在 HomeRobot 中的用法

  • 使用 Habitat 2.0 运行 HSSD 场景数据集中的 OVMM episodes。
  • 提供 Ground Truth 语义分割作为感知上界对比。
  • 用 DDPPO 在 Habitat 中训练 RL 导航/放置策略，然后 sim-to-real 迁移到真实 Stretch。
  • HomeRobot 的 home_robot_sim 子库封装了 Habitat 的 Environment 接口，使其与真实机器人 API 保持一致。

3HSSD 场景数据集

Habitat Synthetic Scenes Dataset (HSSD) 包含 200+ 人工创作的 3D 家庭场景，超过 18,000 个真实世界对象 3D 模型。HomeRobot 使用其中 60 个具有 rearrangement 元数据的场景（训练/验证/测试 = 38/12/10）。场景特点：多房间、有家具遮挡、真实建筑布局，导航和操作难度接近真实世界。

4仿真局限

⚠️ Magic Snap 问题

Habitat 当前版本不物理模拟 gripper-object 接触。HomeRobot 的仿真抓取用"magic snap"——只要机械臂到达对象附近一定距离，对象就自动吸附到末端执行器。这是论文明确承认的局限，导致真实抓取成功率无法从仿真中准确预测。

Stretch Robot Platform

Hello Robot Stretch · 低成本合规移动操作平台

📌 TL;DR

Stretch 是 Hello Robot 公司推出的低成本（$25,000）、人形安全的移动操作机器人，已部署在 40+ 大学和工业实验室。HomeRobot 选它作为真实世界基准平台，正是因为其可复制性——任何实验室都能买到并运行同样的实验。

1硬件规格

  • 底盘：差分驱动，能在室内自由导航。
  • 操作臂：竖向伸缩柱 + 水平臂 + DexWrist，提供 6 自由度操作能力。
  • 传感器：头部 RGB-D 摄像头（用于感知和 SLAM），2D 激光雷达（用于导航）。
  • 计算：机器人本体运行轻量 ROS 节点，感知和规划卸载到外部 GPU 工作站。

2两种工作模式

模式摄像头朝向适用场景
Navigation mode朝前（行进方向）导航探索，使用响应式低层控制
Manipulation mode朝向机械臂抓取/放置，底盘横向平移充当额外自由度

这一设计意味着机器人不能同时看路走路和操作——需要显式切换模式，这增加了系统复杂度但降低了硬件成本。

3在 HomeRobot 中的软件栈

真实 Stretch 运行 home_robot_hw 子库，包括：Hector SLAM 获取全局位姿、ROS 低层控制、与桌面 GPU 工作站的 ROS 通信。所有高层策略代码（感知、规划）在桌面端运行，通过消费级路由器（NetGear Nighthawk）与机器人交互。

4可复制性意义

🔑 为什么选 Stretch

$25,000 的价格在机器人研究中属于相对可负担——比 Boston Dynamics Spot（约 $75,000）便宜，比工业机械臂系统便宜很多。在论文发布时已有 40+ 实验室使用，是 HomeRobot "可复现基准"目标的硬件基础。

Sim-to-Real Transfer

Simulation-to-Real Transfer · 仿真到真实世界迁移

📌 TL;DR

Sim-to-real 是指在仿真中训练的策略能够直接（或通过少量调整）在真实机器人上工作的能力。HomeRobot 的核心基础设施目标之一就是最小化仿真和真实之间的"鸿沟"，通过统一 API 实现导航和放置技能的 sim-to-real 迁移。

1什么导致 Sim-to-Real 失败

  • 视觉 domain gap：仿真图像与真实图像外观差异（光照、纹理、噪声）。
  • 物理 domain gap：仿真中接触模型不准确（尤其是抓取）。
  • API 不一致：仿真和真实机器人使用不同接口，需要大量适配工程。
  • 传感器噪声：仿真 SLAM 完美，真实 SLAM 有漂移和噪声。

2HomeRobot 的应对方案

  • 统一 API：home_robot 核心库定义相同接口，home_robot_sim 和 home_robot_hw 分别实现——策略代码不需要修改就能在两个后端上运行。
  • 使用真实感知模型：仿真中也可以使用 DETIC（而不只用 GT 分割），减少视觉 domain gap。
  • 连续动作空间：避免离散动作空间在真实机器人上的不自然表现。

3HomeRobot 的 Sim-to-Real 结果

论文报告导航和放置技能有 sim-to-real 迁移证据，真实世界 RL baseline 达到 20% 总体成功率。但论文措辞谨慎——"show evidence of sim-to-real transfer"，不是"完全解决了"。抓取的 sim-to-real 迁移因为 magic snap 问题仍是最大挑战。

4与后续 VLA 的关系

🔗 对 Track A 的意义

HomeRobot 提供的 sim-to-real 基础设施是后续 VLA 工作的评测环境。当 π0.5 等 VLA 在 "真实家庭" 中评测时，其所使用的基础设施（Habitat 仿真预训练 + Stretch 真实评测）正是 HomeRobot 所定义的。理解 sim-to-real gap 的来源有助于理解 VLA 为何需要大量真实数据。

HSSD Scene Dataset

Habitat Synthetic Scenes Dataset · Habitat 合成场景数据集

📌 TL;DR

HSSD 是 HomeRobot 仿真基准使用的 3D 场景数据集，包含 200+ 人工创作的真实感家庭场景和 18,000+ 对象模型。与程序化生成的场景（如 ProcTHOR）不同，HSSD 场景由人工设计，布局更自然、更接近真实家庭。

1数据集规格

  • 200+ 人工创作的 3D 家庭场景（多房间、真实建筑布局）。
  • 18,000+ 真实世界对象的 3D 模型（从 AI2-Thor、Amazon-Berkeley Objects、Google Scanned Objects 等来源聚合）。
  • 为 HomeRobot 标注了 2,535 个对象，129 个类别，21 种接收器类别。
  • HomeRobot 使用其中 60 个有 rearrangement 元数据的场景（38 训练 / 12 验证 / 10 测试）。

2对象分类协议

类型训练集验证/测试集
SC, SI（已见类别，已见实例）用于训练 episode不出现
SC, UI（已见类别，未见实例）不用于训练 episode验证/测试
UC, UI（未见类别，未见实例）不用于训练 episode验证/测试

这一划分确保测试时的对象实例（甚至类别）在训练时从未出现，真正考验 open-vocabulary 能力。

3与程序化场景的对比

ProcTHOR 等工作使用程序化生成场景，理论上可生成无限场景，但布局往往不自然。HSSD 牺牲了数量换取质量——人工创作的场景更像真实公寓，但每个场景的制作成本更高。HomeRobot 认为这种"质量换数量"的权衡更有利于评测真实家庭助手能力。

Baseline Agents Modular vs RL

HomeRobot 启发式基线 vs. 强化学习基线

📌 TL;DR

HomeRobot 提供两类 baseline：启发式（模块化）使用运动规划 + 预定规则，RL 用 DDPPO 在 Habitat 中训练深度+分割策略。两者都通过 OVMMAgent 状态机调用相同的 FindObj/Gaze/Pick/FindRec/Place 技能序列。

1共同架构：OVMMAgent 状态机

两类基线都通过同一个高层策略 OVMMAgent 驱动，OVMMAgent 是一个简单状态机，按顺序调用技能：

FindObj → Gaze → Pick → FindRec → Place

区别仅在于 FindObj、Gaze、FindRec、Place 这四个技能的实现方式（启发式 vs. 学习型）。

2启发式基线

  • FindObj / FindRec：DETIC 提供对象分割 → 构建 2D 语义地图 → Fast Marching Method 运动规划。
  • Gaze：基于规则的头部/底盘调整，让对象居中在视野中。
  • Pick：启发式抓取策略（俯视抓取为主，对象较远时侧面抓取）。
  • Place：基于规则的运动到目标接收器上方放置。
  • 优势：2D 语义地图能聚合有噪声的 DETIC 预测，对 DETIC 误分类有一定鲁棒性；更好的长时域探索。

3RL 基线

  • 使用 DDPPO（Decentralized Distributed PPO）在 Habitat 仿真中训练四个技能策略。
  • 输入：深度图 + Ground Truth 语义分割（训练时）/ DETIC 分割（测试时） + 本体感知（关节状态、夹爪状态）。
  • 注意：RL 策略不直接使用 RGB 图像，依赖分割结果作为间接视觉输入。
  • 优势：对象可见时移动更高效；RL 放置技能精度更高（对象更靠近接收器表面）。
  • 劣势：对 DETIC 误分类更敏感（没有语义地图的聚合缓冲）。

4关键发现

💡 感知 >> 算法选择

使用 GT 分割时：RL 全配置总体 SR = 14.8%，启发式全配置 = 5.1%。使用 DETIC 时：所有配置都降至 0.4–0.6%。感知误差的影响是算法选择影响的 10-30 倍——这是 HomeRobot 最重要的实验发现，直接指出 open-vocabulary 感知是领域最需要突破的瓶颈。

DETIC Open-Vocabulary Detector

Detecting Twenty-Thousand Classes using Image-Level Supervision · 开放词汇目标检测器

📌 TL;DR

DETIC 是 Meta AI Research 的开放词汇目标检测器，能够检测 20,000+ 类别的对象（远超 COCO 的 80 类），是 HomeRobot 中启发式和 RL baseline 都使用的感知前端。但它在 OVMM 场景下的误分类率较高，成为性能的主要瓶颈。

1工作原理

DETIC 利用图像级监督（ImageNet 21K 分类标签）训练检测器，通过 CLIP 等视觉-语言对齐将词汇扩展到任意文本描述的类别。关键创新是用弱监督（图像分类标签）代替昂贵的边界框标注来训练检测头。

2在 HomeRobot 中的用法

  • 对 RGB 图像运行推理，为指定类别名（start_receptacle、object、goal_receptacle）提供像素级分割掩码。
  • 启发式基线用分割掩码更新 2D 语义地图。
  • RL 基线在测试时用 DETIC 替换训练时使用的 GT 分割（domain gap 来源之一）。

3为什么 DETIC 成为瓶颈

OVMM 的特殊挑战在于对象类别是动态指定的（任意语言名称），且对象尺寸小、场景杂乱。DETIC 训练数据与机器人视角的 domain gap 导致：

  • 小对象（杯子、药瓶）频繁漏检或误分类。
  • 不同光照/视角下同一对象识别不稳定。
  • 相似外观对象间的混淆（如"茶壶"vs."水壶"）。

这导致仿真中从 GT 分割切换到 DETIC 时性能从 ~15% 崩至 0.4%，差距超过 30 倍。

4改进方向

🔮 论文指出的方向

论文建议未来可用 CLIP-Fields、ConceptFusion、USA-Net 等全开放词汇场景表征替换 DETIC，这些方法将场景表示为连续的语义场而非离散类别掩码，对词汇的泛化能力更强。这也是后续 VLA 工作（直接从 RGB + 语言端到端预测动作，绕过显式语义分割）的重要动机。
