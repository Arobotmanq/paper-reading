Embodied Agent 具身智能体

Embodied Robotic Agent · 具身机器人智能体

📌 TL;DR

Embodied Agent 是一个在物理世界中「感知→推理→行动」闭环运作的 AI 系统。与纯软件 AI agent 不同，它必须与真实环境交互，因此受物理限制（重力、惯性、传感器噪声）约束。Being-0 是一个部署在全身人形机器人上的 embodied agent。

1定义

Embodied Agent 具备三要素：

  • 感知（Perception）：通过传感器（相机、IMU、触觉）观测环境。
  • 推理（Reasoning）：根据感知结果和任务指令决策下一步行动。
  • 行动（Action）：通过物理执行器（电机、关节）改变环境状态。

「Embodied」的核心含义是：智能体的认知和行为不可分割地与其身体和环境耦合——相机位置、关节状态、重力都影响决策。

2本文用法

Being-0 是一个部署在 Unitree H1-2 人形机器人上的 embodied agent，执行「make coffee」「deliver basket」等真实世界长程任务。它通过双目相机和 IMU 感知，用 FM + Connector 推理，通过 41 DoF 关节动作与环境交互。

3对比

类型感知行动典型例子
软件 AI Agent文本/图像（数字）调用 API、生成文本ChatGPT plugins、AutoGPT
Embodied Agent相机/IMU/触觉（物理）控制电机关节（物理）Being-0、RT-2、SayCan

4出处

📜 背景

「Embodied AI」这一概念源于认知科学的 embodied cognition 理论（Lakoff & Johnson 1999），主张认知不可脱离身体和环境而独立存在。在机器人领域，Firoozi et al. 2023 的综述《Foundation Models in Robotics》对 embodied agent 的现状和挑战有全面梳理，Being-0 的 §6 Related Work 直接引用了它。

Foundation Model 基础模型

Foundation Model (FM) · 大规模预训练基础模型

📌 TL;DR

Foundation Model 是在海量互联网数据上预训练的大型多模态模型（通常为 VLM/LLM），具有通用的视觉-语言推理能力。Being-0 用 GPT-4o 作为高层规划者，负责理解指令、判断进度、选择下一个技能——但它不直接控制机器人关节。

1定义

Foundation Model（Bommasani et al. 2021 首提此术语）特指：

  • 在极大规模数据上预训练（图文/代码/视频等）；
  • 具备广泛的任务泛化能力，无需针对每任务重新训练；
  • 通过提示（prompting）或微调适配下游任务。

在机器人语境下，FM 通常指能做视觉理解 + 语言推理的大型 VLM，如 GPT-4V/4o、Gemini、Claude。

2本文用法

Being-0 用 GPT-4o 作为 FM，部署在云端，执行三项功能（§3.2）：

  • Reasoning：描述当前图像，理解任务进度和当前阶段。
  • Detection：判断最近执行技能是否成功，识别异常。
  • Planning：基于推理和检测结果，从技能库中选择下一个技能。

FM 只在关键决策点触发（如技能完成后、任务阶段切换时），不参与每步控制——这是系统效率的关键。

3对比：FM vs Connector VLM

Foundation Model (GPT-4o)Connector VLM (VideoLLaMA2)
规模极大（未知，估计千亿参数）轻量（数十亿参数）
部署位置云端 API板载 Jetson AGX
训练数据互联网通用数据室内导航第一人称图像（领域专用）

4出处

📜 起源

「Foundation Model」术语由 Stanford HAI 的 Bommasani et al. 2021 在《On the Opportunities and Risks of Foundation Models》中系统提出。在机器人 agent 中使用 FM 的代表性工作包括 SayCan（Ahn et al. 2022，Google）和 Cradle（Tan et al. 2024）——Being-0 明确基于 Cradle 框架改造而来。

Connector 连接器模块

Embodied Connector · 具身连接器 (VLM-based 中间层)

📌 TL;DR

Connector 是 Being-0 最核心的创新——一个用室内导航数据微调的轻量 VLM（VideoLLaMA2），专门负责把 FM 的语言计划实时翻译为可执行技能命令，并通过 Grounded Skill Planning + Visual Navigation + Pose Adjustment 三项功能弥合 FM 与低层技能之间的「感知-执行鸿沟」。

1为什么需要 Connector

直接把 FM（GPT-4o）连接到技能库时，三个问题导致系统失效：

  • FM 推理太慢：GPT-4o 延迟数秒，无法以所需频率（约 1 Hz）闭环控制双足机器人；
  • FM 3D 空间理解差：GPT-4o 无法准确估计目标物体的方向和深度，导致 locomotion 方向规划频繁出错；
  • 导航终止位姿不对：即使导航到达目标附近，停止位姿可能不适合后续抓取（物体在抓取手背后、距离太远等）。

2三项核心功能

  • Grounded Skill Planning：把 FM 的语言计划（如「抓杯子」）映射到当前可执行技能（如「向桌子移动」），并在 FM 计划与当前状态不匹配时自动纠正/细化。
  • Visual Navigation：用目标物体的检测边界框 + 双目合成深度估计相对位置，实时选择 joystick locomotion 技能（直走/转向等）闭环导航，遇到目标不可见时触发探索程序（转头+移动搜索）。
  • Pose Adjustment：在即将到达目标时，预测最优对齐方向，触发「弧形接近」复合技能，使机器人以适合操作的位姿终止导航——这是 Grasp-coffee 成功率从 1/5 提升到 4/5 的关键。

3训练方式

以 VideoLLaMA2 为 backbone，在室内导航第一人称图像数据集上做多任务微调（Multi-task Learning），三个训练目标并行：

  • 图像描述（image description）；
  • 技能预测（skill prediction）；
  • 目标检测（object detection，输出边界框）。

训练后部署在板载 Jetson AGX 上，推理延迟约 1 秒，显著低于 GPT-4o 的云端延迟。

4类比

🎯 类比

可以把 Connector 类比为「公司里层经理」：FM（高管）下达战略指令（「搞定这个项目」），Connector（中层经理）把指令分解成当前可执行的具体工作任务，并实时监控进度、动态调整——不是每件小事都上报高管，而是在权限范围内自主处理。只有阶段性节点才需要 FM 介入判断。

Modular Skills 模块化技能库

Modular Skill Library · 模块化技能库

📌 TL;DR

Modular Skill Library 是一组独立、可复用的低层控制策略，每个技能对应一个具体动作（「直走」「抓杯子」「按按钮」），通过语言描述接口被 Connector 和 FM 调用。上下体技能完全解耦——下体 locomotion 用 RL 训，上体 manipulation 用遥操+行为克隆训。

1定义

模块化技能库是一组预训练的、功能单一的策略集合，每个技能：

  • 有明确的语言描述（如 grasp_cup、go_straight）；
  • 有固定的输入/输出接口（观测 → 关节目标位置）；
  • 可被上层系统通过名称独立调用，无需了解内部实现。

2本文的两类技能

技能类型控制部位训练方法控制频率接口
Locomotion Skills下体（13 DoF 腿+躯干）RL + sim-to-real50 Hzjoystick velocity command
Manipulation Skills上体（7×2 DoF 手臂 + 6×2 DoF 手）遥操数据 + ACT 行为克隆10 Hz语言描述（如 "grasp bottle"）

3可扩展性

模块化设计的核心优势：

  • 新操作技能只需 50–150 条遥操轨迹，<1 小时采集；
  • locomotion 技能更换不影响 manipulation 策略，反之亦然；
  • 已扩展到触觉传感器增强的灵巧手（Play-chess 任务）。

4与 Whole-body Control 的对比

Modular Skills（Being-0）Whole-body Control（如 HumanPlus）
上下体关系完全解耦，独立策略单一策略同时控制上下体
技能多样性易扩展（独立训练每个技能）难（需要重新训练整体策略）
运动+操作同时仅支持静止时操作支持行走同时操作

VLA 视觉-语言-动作模型

Vision-Language-Action Model · 视觉-语言-动作模型

📌 TL;DR

VLA 是一类直接把视觉观测 + 语言指令端到端映射到机器人动作的模型（图像 + 文字 → 关节目标位置/速度）。在 Being-0 中，VLA 的概念对应底层的 manipulation skill（用 ACT 行为克隆实现）——注意：高层的 FM 和 Connector 是 VLM，不是 VLA，它们不直接输出机器人动作。

1定义

VLA = Vision-Language-Action Model，核心特征：

  • 输入：视觉观测（图像/视频）+ 自然语言指令；
  • 输出：机器人动作（关节角度/速度/末端轨迹等）；
  • 端到端：感知和控制在同一个神经网络内完成，不需要手工设计的中间表示。

代表性工作：RT-2（Brohan et al. 2023, Google DeepMind）、π0（Black et al. 2024, Physical Intelligence）、GR-2、OpenVLA 等。

2本文中「VLA」的准确含义

Being-0 的架构中，术语使用需要仔细辨别：

组件实际技术是否是 VLA？输出
Foundation Model (GPT-4o)Large VLM不是 VLA语言文字（技能名）
Connector (VideoLLaMA2)Fine-tuned VLM不是 VLA技能命令/边界框
Manipulation Skill (ACT)行为克隆 Transformer近似 VLA（图像+语言→动作）关节目标位置（上体）
Locomotion Skill (RL policy)RL 策略不是 VLA（输入仅 proprioception）关节目标位置（下体）

论文 Related Work §6 明确将 VLA 作为独立方向提及（RT-1/RT-2/π0 等），但 Being-0 选择不走「端到端 VLA」路线，而是用「FM + Connector + 模块化 skills」的分层架构——核心原因是缺乏人形机器人灵巧手的大规模数据集。

3类比

🎯 对比与类比

VLA 相当于「一个端到端训练的全能运动员」，而 Being-0 的做法是「教练（FM）+ 助理教练（Connector）+ 专项运动员（各 skill）」的分工体系。前者目标更优雅，但需要大量覆盖全分布的示范数据；后者牺牲一些全局最优性，换来可扩展性和当前数据条件下的实用性。

Humanoid Locomotion Policy 人形运动策略

Humanoid Locomotion Policy · 人形机器人运动控制策略

📌 TL;DR

Humanoid Locomotion Policy 是控制双足机器人稳定行走的神经网络策略，通常用强化学习在仿真中训练，再做 sim-to-real 迁移到真机。在 Being-0 中它是技能库中的一个组件，通过 joystick velocity command 接口被 Connector 调用。

1定义

Locomotion Policy 是一个映射 \(\pi_L(a_l \mid q_l, q_u, \dot{q}, \omega; v_g)\) 的神经网络策略：

  • 输入：下体/上体关节位置 \(q_l, q_u\)、关节速度 \(\dot{q}\)、IMU 角速度 \(\omega\)、目标 joystick 速度 \(v_g\)；
  • 输出：下体关节目标位置 \(a_l\)（PD 控制器跟踪）；
  • 控制频率：50 Hz。

2本文训练方式

Being-0 采用RL + sim-to-real：

  • 在 Isaac Gym 等物理仿真器中用强化学习训练 goal-conditioned 策略；
  • 加入 domain randomization（地面摩擦、机器人质量分布等随机化）和外力干扰，增强鲁棒性；
  • 直接迁移到真机（zero-shot sim-to-real）。

技能库中定义了 9 种 locomotion skill：no action、go straight、walk backwards、turn left、turn right、sidestep left、sidestep right，以及头部控制的 tilt head、turn head。

3与 Track B 的关系

Being-0 的 locomotion skill 是功能型组件，不是 locomotion 算法的研究贡献。论文引用 Ha et al. 2024《Learning-based Legged Locomotion: State of the Art and Future Perspectives》作为方法来源，并承认当前仅支持平地行走，不支持复杂地形、蹲坐跳等——Track B 关注的高级 locomotion 算法（如 parkour、楼梯、多地形）均不在本文讨论范围内。

4类比

🎯 类比

Locomotion Policy 相当于「会走路的本能反射」——人类走路时不需要有意识地计算每步如何迈脚，大脑只需发出「向左走」的高层指令。Being-0 的 locomotion skill 正是这样：Connector 发出「turn_left」命令，策略自动处理所有低层关节协调，保持平衡。

RL Locomotion 强化学习运动控制

Reinforcement Learning for Locomotion · 基于强化学习的运动控制

📌 TL;DR

RL Locomotion 用强化学习让机器人在仿真中通过「试错」学会行走，再部署到真机。相比传统最优控制（MPC/LQR），RL 方法能自动学到对不规则地形、外力干扰的鲁棒策略，无需手工设计状态机或步态参数。Being-0 的 locomotion skill 正是 RL + sim-to-real 路线的产物。

1定义

强化学习运动控制的基本范式：

  • 奖励函数：定义好的行走目标（跟踪速度指令 + 保持直立 + 能量效率 - 跌倒惩罚）；
  • 仿真环境：Isaac Gym（NVIDIA）、MuJoCo 等，支持并行化大批量训练；
  • 策略网络：通常是 MLP 或小型 Transformer，接受 proprioception 输出关节目标；
  • sim-to-real 迁移：通过 domain randomization 缩小仿真与真实的物理差异。

2本文中的具体做法

Being-0 采用 Ha et al. 2024 的 RL locomotion 框架，训练一个以 joystick 速度命令 \(v_g\) 为条件的策略，支持多方向行走同时保持平衡。仿真中加入了 domain randomization 和 external forces，提升真机部署的鲁棒性（§3.1）。

3与传统方法的对比

方法核心思路优点缺点
最优控制（MPC/LQR）数学建模 + 在线求解优化理论保证、可解释需精确模型、计算量大
RL + sim-to-real仿真中试错学习，迁移真机无需精确模型、鲁棒性强奖励设计难、sim2real gap
模仿学习学习运动捕捉数据动作自然需要 mocap 数据

4出处

📜 代表性工作

RL Locomotion 在人形机器人上的代表性进展：Xie et al. 2020（平地行走）、Radosavovic et al. 2024（复杂地形，用 transformer policy）、Zhuang et al. 2024（parkour）。Being-0 直接引用 Ha et al. 2024 作为其 locomotion 组件的基础，而不是从头开发 locomotion 算法。

ACT 行为克隆 Transformer

Action Chunking with Transformers (ACT) · 基于 Transformer 的动作分块行为克隆

📌 TL;DR

ACT（Zhao et al. 2023）是一个用 Transformer 做行为克隆（Behavior Cloning）的操作策略，核心思想是每次预测一个「动作块」（action chunk，多步动作序列）而不是单步动作，减少在高频控制中的误差累积。Being-0 用 ACT 训练每一个操作技能（如「抓瓶子」「按按钮」）。

1定义

ACT 的核心设计：

  • 行为克隆（Behavior Cloning）：直接模仿人类遥操示范数据，最小化预测动作与示范动作的 L2 误差；
  • Action Chunking：预测未来 K 步动作序列（chunk），而非单步动作，减少在决策过渡处的抖动；
  • Transformer 架构：利用注意力机制处理多摄像头图像 + 关节状态，生成动作序列。

2本文中的训练配置

Being-0 用 ACT 训练每个 manipulation skill（§3.1）：

  • 输入：双目图像 \(o_l, o_r\)、上体关节位置 \(q_u\)、颈部关节位置 \(q_h\)；
  • 输出：K 步的上体+颈部关节目标位置 \([a_u, a_h]_t^{t+K}\)；
  • 训练时 K=30，部署时 K=10（更保守，允许更频繁重规划）；
  • 每个技能独立训练，数据来自 Apple VisionPro 遥操采集，50–150 条轨迹，<1 小时。

3遥操采集流程

Using Apple VisionPro：操作者佩戴 VisionPro，双目图像投影到 VR 视野，人的头部/手腕/手指动作实时重定向（retarget）到机器人的颈部/手腕/手指关节——这使得操作者能以直觉方式控制机器人完成精细操作，采集高质量示范数据。

4出处与对比

📜 起源

ACT 由 Zhao et al. 2023（Stanford Tony Zhao）提出，在 ALOHA 双臂操作平台上验证，能完成插充电线、打开零食等精细操作。相比 Diffusion Policy（Chi et al. 2023）的生成式动作分布建模，ACT 更轻量，训练更快，适合快速扩展技能库。Being-0 优先考虑了可扩展性（新技能<1小时）而选用 ACT。

Joystick Command 摇杆速度指令

Joystick Velocity Command · 摇杆速度指令接口

📌 TL;DR

Joystick Command 是 locomotion skill 的高层接口——把「向前走多快、转多快」等高层意图编码为速度向量 \(v_g\)，locomotion 策略则负责把它实时转换为 50 Hz 的关节目标位置。Connector 正是通过发送 joystick command 来驱动机器人导航，无需关心底层关节控制细节。

1定义

Joystick Command（也叫 velocity command）是一个目标速度向量 \(v_g = (v_x, v_y, \omega_z)\)：

  • \(v_x\)：前进/后退速度（m/s）；
  • \(v_y\)：左右侧移速度（m/s）；
  • \(\omega_z\)：偏航角速度（rad/s，转向）。

Locomotion policy 以此为条件输入，实时输出满足速度命令同时保持平衡的关节目标位置。

2本文中的离散化

Being-0 不直接发连续的 \(v_g\)，而是把 joystick command 离散成 9 个命名技能（§3.1）：

  • 运动：no action、go straight、walk backwards、turn left、turn right、sidestep left、sidestep right
  • 视觉调整：tilt head、turn head

Connector VLM 预测的是这些离散技能名（分类问题），而不是连续速度值——这降低了 Connector 的输出空间复杂度，适合轻量 VLM 处理。

3为什么用 joystick command 接口而非直接控制关节

抽象层次的分离是关键设计原则：

  • 安全性：locomotion policy 内部处理平衡约束，上层不会直接发出不安全的关节命令；
  • 简洁性：Connector 只需输出 9 类离散命令，不需要理解 13 DoF 下体动力学；
  • 可替换性：底层 locomotion policy 升级不影响 Connector 的工作方式。

4类比

🎯 类比

Joystick Command 接口就像汽车的「油门+方向盘」——司机（Connector）只需控制加速减速和转向方向，不需要直接控制发动机气缸或悬挂系统（低层 locomotion policy 处理这些）。这个接口抽象使得驾驶变得可能，否则要同时处理几百个物理参数。
